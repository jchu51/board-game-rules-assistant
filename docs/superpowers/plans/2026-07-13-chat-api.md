# Chat API Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add create, paginated list, paginated detail, and delete chat endpoints backed consistently by memory and Postgres persistence.

**Architecture:** Extend the conversation repository into the source of truth for chat metadata and messages. A `ChatService` validates lifecycle outcomes while `ChatRouter` owns HTTP parsing and response schemas. Cursor codecs remain internal helpers and both persistence drivers implement identical contracts.

**Tech Stack:** TypeScript, Express, Zod, Vitest, PostgreSQL, OpenAPI 3.1

## Global Constraints

- Implement the approved contract in `docs/api-designs/chat-api/api-design.md`.
- Keep existing retrieval clients compatible when they append messages to an unknown conversation ID.
- Bound chat and message reads with opaque cursor pagination.
- Preserve the current `{ "error": string }` error envelope and unversioned route convention.
